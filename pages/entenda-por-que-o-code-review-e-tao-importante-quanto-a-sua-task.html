Entenda por que o code review é tão importante quanto a sua task

Neste post, compartilho conhecimentos sobre tecnologias que podem contribuir para o desenvolvimento de projetos escaláveis.

Neste post, vou falar sobre um tema que muitos desenvolvedores já enfrentaram – ou estão enfrentando agora –, especialmente quem é novo na empresa ou sente aquela pressão de "mostrar serviço". Mas calma... não é bem assim que as coisas funcionam! Na verdade, podem ser muito mais complicadas do que você imagina. Vou tentar dar uma clareada no assunto, pois sei que muita gente ainda se sente perdida. Vamos lá!

Quando falamos de code review, estamos falando de passar horas analisando o código dos colegas e revisando o seu próprio, o que, para muitos desenvolvedores, é quase um pesadelo, mas o "pesadelo" não é ter que fazer code review em si, é parar o que você está fazendo (sua task) para revisar o código dos coleguinhas (outros times), e esse é um dos principais motivos pelos quais muitos desenvolvedores priorizam suas tasks, do que param o que está fazendo para realizar code review, incluindo desde os Juniors até os Seniors.

O Dilema de Code Review x Task

Vamos falar de uma situação que acontece o tempo todo: você está com uma task em desenvolvimento, importante (visibilidade) ou não, que precisa ser entregue com qualidade, vou listar alguns requisitos comuns que você já deve conhecer no que você neste momento está preocupado:

  • Manter o código limpo
  • O código com os princípios SOLID(s) bem definidos
  • Com testes unitários bem realizados cobrindo todo o cenário
  • Ter mapeado todos os cenários
  • Com possibilidade de voltar seu código umas duzentas mil vezes
  • Com possibilidades do seu coleguinha pegar um erro que você não tinha visto

Só para cumprir esses pontos, já sabemos que leva um bom tempo, né? E, além de tudo isso, ainda tem a possibilidade do seu gestor ou PO perguntando quando você vai terminar a task, e isso é normal, faz parte do trabalho, mas tem um detalhe essencial que ainda não mencionei: o code review.

Aí vem o dilema: você prioriza a sua task ou para tudo para fazer code review? Agora você deve estar pensando: "Claro que eu faço o code review!" Mas faz mesmo? Ou espera o feedback dos colegas para só então começar a revisar com frequência? Pode ser que você já saiba a resposta.

Code review não é opcional

Se você está decidindo entre fazer uma coisa ou outra, já está no caminho errado, porque o code review não é opcional, ele precisa ser parte do seu trabalho, assim como desenvolver melhorias, corrigir bugs e alinhar com o time, ou seja, o code review deve fazer parte do seu tempo, e ponto final.

Ninguém te disse que, ao entrar na empresa, o code review deveria ser parte da rotina? Pois é, não precisa te contar, isso já faz parte do pacote, e assim como você precisa de um tempo para tomar um café ou beber água, você também precisa de um tempo para o code review, e se a sua desculpa for "isso vai atrasar minha entrega", então talvez tenha faltado planejar melhor o tempo para o code review.

O que eu entendi, e que você também deveria entender, é que o code review pode sim, melhorar seu código, ajudar a detectar problemas comuns e reforçar boas práticas, então, aqui estão alguns benefícios do code review:

  • Melhora a qualidade do seu código
  • Ajuda a manter a cultura do code review no time
  • Facilita para que seus colegas vejam pontos que talvez você não enxergou
  • Garante que o SOLID e outras boas práticas estejam presentes
  • Ajuda a empresa a manter um código mais limpo e eficiente
  • Te ajuda a policiar e manter as melhores práticas

É claro que o code review pode parecer uma "perda de tempo" quando estamos correndo para entregar algo, mas, na verdade, ele previne retrabalho no futuro, e as grandes empresas sabem bem disso.

Code review em grandes empresas

Se você está preocupado com o tempo "gasto" com code review (que, na verdade, é um investimento), então vamos a alguns dados importantes:

  • Em algumas grandes empresas, como Microsoft e Google, entre 10% e 20% do tempo de desenvolvimento é dedicado ao code review, em equipes que lidam com código crítico, esse percentual pode chegar a 30%.
  • Um code review eficaz leva entre 30 e 60 minutos e, para garantir a qualidade, o ideal é revisar de 200 a 400 linhas de código por hora.
  • Para facilitar as revisões, essas empresas preferem pull requests menores, de 200 a 400 linhas, evitando aqueles códigos gigantescos e difíceis de revisar.

Estudos mostram que times que investem tempo em code reviews reduzem o número de bugs em produção entre 30% e 50%, o que acaba economizando tempo e custos no longo prazo.

Conclusão: Por que tudo isso?

Eu precisei fazer toda essa contextualização para chegar a um ponto importante: o tempo investido em code review é para o seu benefício e para a empresa, porque tempo é dinheiro!

Dessa forma, você previne bugs e refatorações futuras, evitando retrabalho em algo que poderia ter sido resolvido com a ajuda dos seus colegas, e esse tempo liberado pode ser usado para criar novas funcionalidades em vez de corrigir problemas antigos.

Esse tempo dedicado ao code review, é para o time todo, manter essa cultura de code review, pois quanto mais o time adota a cultura de code review, mais se fortalecem as boas práticas e a qualidade do código, e quando todos entendem o valor disso, o code review deixa de ser um fardo e se torna uma rotina natural que faz parte do seu dia a dia como dev.

Não espere o feedback chegar para começar a fazer code reviews, comece agora mesmo!
